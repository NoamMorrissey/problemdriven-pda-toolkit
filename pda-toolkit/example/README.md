# Example: TeamTasks — Shared Task List for Small Teams

This is a complete, end-to-end example of the Problem-Driven AI methodology applied to a real (fictional) project. Every artifact was created following the PDA process: research first, then problem definition, then solution design, then AI-assisted construction.

## What's Inside

```
example/
├── research/                    ← Phase 1: The evidence
│   ├── interviews/              ← 3 user interviews
│   ├── analytics/               ← Survey data (N=127)
│   └── benchmarks/              ← Competitor analysis (5 tools)
│
├── docs/
│   ├── pda-problem.md           ← Problem Statement (9 elements, Gate 1 PASS)
│   └── pda-solution-brief.md    ← Solution Brief (8 sections, Gate 2 PASS)
│
├── .specify/                    ← Phase 3: Spec Kit artifacts
│   ├── memory/constitution.md   ← Generated by PDA adapter
│   └── specs/001-teamtasks/     ← Spec, plan, tasks (generated)
│
├── pda-verification/            ← Verification reports (all PASS)
│
└── src/                         ← Phase 4: Built application
    └── index.html               ← Functional to-do app (open in browser)
```

## How to Walk Through It

**Start with the research.** Read the three interviews and the survey data. This is the evidence that everything else is built on.

**Then read the Problem Statement.** Notice how every claim references specific evidence. Run `/pda.validate-problem` to see the agent verify it.

**Then read the Solution Brief.** Notice how every component traces back to the Problem Statement via `gap_ref` and `success_ref`. Run `/pda.validate-solution` to see the agent verify it.

**Then look at the Spec Kit artifacts.** These were generated from the Solution Brief through the PDA adapter. The verification reports show what the agents checked.

**Finally, open `src/index.html`.** That's the built application. It does exactly what the Solution Brief described, nothing more.

## The Point

The app itself is simple. That's intentional. The value is not the app — it's the process that got there. Every line of code traces back through tasks → spec → Solution Brief → Problem Statement → evidence from real (fictional) people.

That's what Problem-Driven AI does: it ensures the AI builds the right thing, not just builds things right.
